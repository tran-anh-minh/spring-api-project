`def call_llm(prompt: str, config) -> str | None`:
- If not config.learning.llm_provider or not config.learning.llm_api_key: return None
- try/except ImportError and Exception, returning None on failure:
  - If config.learning.llm_provider == "claude":
    - `import anthropic` (inside function body, not module level)
    - client = anthropic.Anthropic(api_key=config.learning.llm_api_key)
    - msg = client.messages.create(model=config.learning.llm_model or "claude-sonnet-4-20250514", max_tokens=1024, messages=[{"role": "user", "content": prompt}])
    - return msg.content[0].text
  - If config.learning.llm_provider == "openai":
    - `import openai` (inside function body)
    - client = openai.OpenAI(api_key=config.learning.llm_api_key)
    - resp = client.chat.completions.create(model=config.learning.llm_model or "gpt-4o", messages=[{"role": "user", "content": prompt}], max_tokens=1024)
    - return resp.choices[0].message.content
- Log warning on failure with exc_info=True

Import sqlite3, json, logging. Import GapRecord, AgentFindings, FindingItem from models.

`SAMPLING_QUERIES` dict mapping gap_type to a function that generates SQL queries:
- "unlabeled_enum": given entity_name "table.column", generate `SELECT DISTINCT {column} FROM {table} ORDER BY {column}` (parameterized table/column from entity_name split)
- "missing_fk": given "table.column_id", generate `SELECT DISTINCT {column_id}, COUNT(*) FROM {table} GROUP BY {column_id} ORDER BY COUNT(*) DESC`
- "stale_fact": same as unlabeled_enum (re-sample current values)
- "low_confidence_fact": same as unlabeled_enum
- Other types: return None (not solvable by sampling)

`def collect_evidence(conn: sqlite3.Connection, gap: GapRecord, config) -> AgentFindings`:
- If config.database.connection_string is None: return AgentFindings(summary="No live DB connection; skipping data collection", evidence_quality=0.0)
- Determine sampling queries for gap.gap_type from SAMPLING_QUERIES
- If no queries for this type: return AgentFindings(summary=f"No sampling strategy for gap type {gap.gap_type}", evidence_quality=0.0)
- For each query (up to config.learning.collector_query_budget):
  - Call _execute_sample(config.database.connection_string, query, config.learning.collector_timeout_seconds, config.learning.collector_max_rows)
  - Convert results to FindingItem list
- Return AgentFindings(items=items, summary=f"Collected {len(items)} samples", evidence_quality=0.5 if items else 0.0)

`def _execute_sample(connection_string: str, query: str, timeout: int, max_rows: int) -> list[dict] | None`:
- try: import pyodbc (inside function body)
- except ImportError: return None
- If not connection_string: return None
- try:
  - conn = pyodbc.connect(connection_string, timeout=timeout, readonly=True)
  - cursor = conn.cursor()
  - cursor.execute(query)
  - rows = cursor.fetchmany(max_rows)
  - return [dict(zip([col[0] for col in cursor.description], row)) for row in rows]
- except Exception: logger.warning("Collector sampling failed", exc_info=True); return None
- finally: try conn.close() except pass

`def research_gap(conn: sqlite3.Connection, gap: GapRecord, evidence: AgentFindings, config) -> AgentFindings`:
- This is the Investigate+Reason phase. It analyzes the gap using evidence from collector + existing knowledge.
- Step 1: Gather context -- query related entities from the knowledge store using gap.entity_name. Use hybrid_search(conn, gap.entity_name, limit=5) to find related entities.
- Step 2: Try LLM reasoning if configured:
  - Build prompt: "Analyze this knowledge gap in a SQL Server database:\nGap: {gap.gap_type} for {gap.entity_name}\nDescription: {gap.description}\nEvidence collected: {evidence.summary}\nEvidence items: {json.dumps([item.model_dump() for item in evidence.items[:10]])}\nRelated entities: {related_context}\n\nProvide structured findings as JSON array of objects with keys: entity_type, entity_name, attribute, value, confidence (0.0-1.0).\nOnly include findings supported by the evidence."
  - llm_response = call_llm(prompt, config)
  - If llm_response: parse JSON from response, create FindingItem list. Set evidence_quality = 0.7.
- Step 3: If no LLM (offline mode), use heuristic analysis:
  - For "unlabeled_enum": if evidence has sampled values, create FindingItem for each distinct value as potential enum label. confidence = 0.4 (heuristic).
  - For "orphan_table": create FindingItem suggesting description = "Potentially unused table" with confidence 0.3.
  - For "missing_fk": if evidence shows matching ID patterns, suggest fk_inferred relationship. confidence = 0.4.
  - For "coverage_gap": create FindingItem with attribute="description", value="[Auto-generated: requires human review]", confidence = 0.2.
  - For other types: return empty findings with summary "No heuristic available for {gap.gap_type}"
  - Set evidence_quality = 0.3 for heuristic findings
- Return AgentFindings(items=findings, summary=summary_text, evidence_quality=quality)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Live DB -> Collector | Collector queries external database; results are untrusted input |
| LLM response -> Research Agent | LLM output parsed as JSON; must handle malformed responses |
| Agent findings -> Review Agent | Review validates before knowledge store mutations |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08 | S (Spoofing) | LLM response | mitigate | Parse LLM JSON in try/except; reject malformed responses. Never execute LLM output as code. Confidence capped at 0.7 for LLM-derived facts. |
| T-03-09 | T (Tampering) | Collector SQL injection | mitigate | Collector generates SELECT queries from entity_name. Entity names come from knowledge store (not user input). Validate table/column names match [a-zA-Z0-9_] pattern before query construction. Never use f-string SQL for WHERE clauses. |
| T-03-10 | I (Information Disclosure) | LLM API key in logs | mitigate | Never log config.learning.llm_api_key. Logger.warning in call_llm uses exc_info which may include request headers -- ensure anthropic/openai SDKs redact keys in exception messages (they do by default). |
| T-03-11 | D (Denial of Service) | Collector timeout | mitigate | pyodbc.connect(timeout=config.collector_timeout_seconds). cursor.fetchmany(max_rows) limits result size. query_budget limits total queries per run. |
</threat_model>
