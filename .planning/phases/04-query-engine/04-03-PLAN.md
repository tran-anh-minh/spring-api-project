             sub_questions = analyst.decompose(question, tier.value)

              - gen = generate_sql(question, tier.value, ctx.text, config, metrics, previous_error)

              - errors = validate_sql(gen.sql, self._schema_map)
              - If no errors, break (success)
              - previous_error = "; ".join(errors)
           g. Return QueryResult with sql=gen.sql, validation_errors=errors, attempts=attempt_count, context_tokens=ctx.token_count

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pipeline -> live DB | Generated SQL crosses from local system to production database |
| LLM response -> pipeline | LLM-generated SQL or decomposition enters the pipeline |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-06 | Injection | executor.py:execute_query | mitigate | Only SELECT statements allowed -- check sql.strip().upper().startswith("SELECT"). Reject INSERT/UPDATE/DELETE/DROP/EXEC. Row count limited by config.query.max_execution_rows. Connection timeout enforced via config.database.timeout_seconds. |
| T-04-07 | Elevation of Privilege | pipeline.py:run | mitigate | Pipeline does not execute SQL by default -- execution requires explicit execute=True parameter. Read-only safety constraint enforced at executor level. |
| T-04-08 | Tampering | cache.py:query_cache | accept | Cache is local SQLite -- same trust boundary as knowledge store. Poisoned cache entry would return wrong SQL but cannot execute without explicit execute=True. Cache cleared on schema version change. |
| T-04-09 | Spoofing | analyst.py:decompose | accept | LLM decomposition could produce misleading sub-questions, but each sub-query goes through full validation pipeline. Worst case: bad SQL caught by validator, retried, or returned as None. |
</threat_model>
