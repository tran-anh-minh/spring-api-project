<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| NL question -> SQL generation | Untrusted user input influences LLM prompt and template output |
| LLM response -> SQL execution | LLM-generated SQL could contain harmful statements |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Injection | generator.py:build_generation_prompt | mitigate | User question is embedded in prompt as plain text with explicit "Generate T-SQL only" instruction. LLM output always goes through validate_sql() before any execution. |
| T-04-04 | Injection | validator.py:validate_sql | mitigate | All generated SQL must pass sqlglot.parse_one() + qualify(). Only SELECT statements should proceed to execution. The validator catches unknown tables/columns but does NOT check for DDL/DML — the executor (Plan 03) must verify statement type is SELECT before executing. |
| T-04-05 | Information Disclosure | context.py:assemble_context | accept | Context assembly reads from local SQLite knowledge store. No external data exposure. Schema information is intentionally shared with LLM for SQL generation. |
</threat_model>
